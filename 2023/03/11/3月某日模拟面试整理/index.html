    在 JavaScript 中，一个闭包（Closure）是指一个函数及其包含的所有变量的集合体，其中函数内部可以访问包含它的函数的变量，即使包含它的函数已经返回。

    在这个例子中，outerFunction() 定义了一个内部函数 innerFunction()，并返回它。在 innerFunction() 中，我们可以访问 outerVariable，这是因为 innerFunction() 形成了一个闭包，包含了 outerFunction() 的作用域。当我们调用 outerFunction() 并将结果赋值给 closure 后，closure 现在是一个指向 innerFunction() 的引用，它可以在任何时候被调用并输出 outerVariable 的值。即使 outerFunction() 已经返回，outerVariable 仍然存在于闭包中，可以被访问。

    值得注意的是，在 JavaScript 中，闭包是一种非常常见的技术，它可以用来创建函数工厂、模块化代码、保存私有变量等。

​ 在 JavaScript 中，闭包是由一个函数和它周围的引用环境（也称为词法环境）组成的。具体来说，当一个函数被定义时，它会创建一个新的词法环境，该环境包含了该函数内部的所有变量、参数和函数声明。

​ 当该函数返回时，它可能会返回一个内部函数，并将该函数作为一个值赋给外部函数中的一个变量。如果内部函数引用了外部函数中的变量或参数，那么这些变量和参数就会在内部函数和外部函数之间形成一个闭包。这意味着内部函数可以访问外部函数的变量和参数，即使外部函数已经返回并且它们不再存在于调用堆栈中。

​ 在JavaScript中，当一个函数可以访问其外部函数作用域中的变量时，就形成了一个闭包。由于闭包中包含对外部函数作用域中变量的引用，因此这些变量不会被垃圾回收机制回收，直到闭包被销毁。

要销毁闭包，需要先将引用闭包的变量置为null。这样可以使垃圾回收机制可以回收被闭包引用的变量。另外，如果不再需要使用闭包，可以将闭包变量赋值为null，这样闭包所占用的内存就可以被释放。

  • 解释一下作用域这个东西

    • 作用域指的是您有权访问的变量集合。

      在 JavaScript 中有两种作用域类型：

      ​ 局部作用域

      ​ 全局作用域

      ​ JavaScript 拥有函数作用域：每个函数创建一个新的作用域。

      ​ 作用域决定了这些变量的可访问性（可见性）。

      ​ 函数内部定义的变量从函数外部是不可访问的（不可见的）。

    • JavaScript中的作用域是指变量和函数的可访问范围。在JavaScript中，每个函数都有自己的作用域。当函数被调用时，它会创建一个新的作用域，并在该作用域中定义函数的参数、局部变量和内部函数。

      JavaScript中有两种类型的作用域：全局作用域和局部作用域。全局作用域是在函数外部定义的变量和函数，在任何函数内都可以访问它们。局部作用域是在函数内部定义的变量和函数，只能在该函数内部访问。

      当在函数内部引用一个变量时，JavaScript会首先搜索局部作用域以查找该变量。如果找到了该变量，则使用该变量。否则，它会继续在父级作用域中查找变量，直到找到该变量或搜索到全局作用域为止。如果在全局作用域中也没有找到该变量，则该变量未定义，JavaScript将抛出一个错误。

      在ES6之前，JavaScript只有函数作用域。在ES6中，引入了块级作用域。块级作用域是在代码块（例如if语句或for循环）内部定义的变量和函数，只能在该代码块内部访问。使用块级作用域可以避免一些常见的变量共享问题，并使代码更易于维护和理解。

  • webpack是什么

​ Webpack是一个现代JavaScript应用程序的静态模块打包工具，它可以将各种类型的前端资源（如JavaScript、CSS、HTML、图片等）打包成一个或多个静态资源文件，以便于浏览器加载。Webpack可以帮助前端开发者管理模块依赖关系，优化资源文件大小和加载速度，提高开发效率等。

​ Webpack的主要特点包括：

  • 模块化打包：Webpack支持ES6、CommonJS、AMD等多种模块化规范，并可以将它们打包成一个或多个静态资源文件。
  • 智能依赖管理：Webpack可以分析模块之间的依赖关系，自动加载依赖模块，并按需打包，避免重复加载和冗余代码。
  • 丰富的插件机制：Webpack提供了丰富的插件机制，可以通过插件对打包过程进行自定义配置和优化，例如代码压缩、文件合并、图片压缩等。
  • 热模块替换：Webpack支持热模块替换（HMR）技术，可以在不刷新页面的情况下，实时更新应用程序的代码和样式，提高开发效率。
  • 多种输出格式：Webpack可以将打包后的资源文件输出为CommonJS、AMD、ES6模块或全局变量等多种格式，适应不同的开发场景。

总之，Webpack是一个功能强大、灵活、易于使用的前端构建工具，被广泛应用于各种类型的前端项目中。

  • webpack里面有什么

Webpack里面有很多不同的概念和部件，以下是一些Webpack中的重要组成部分：

  1. Entry：Webpack的入口点，指定应用程序的起始点，Webpack会从这里开始构建依赖图。
  2. Output：Webpack的输出目标，指定打包后的文件存储路径和文件名等信息。
  3. Loader：Webpack中的资源加载器，用于将各种类型的文件（如CSS、图片、HTML等）转换为Webpack能够处理的模块。
  4. Plugin：Webpack的插件机制，用于扩展Webpack的功能，实现自定义的构建流程和优化策略。
  5. Mode：Webpack的模式，包括开发模式和生产模式，用于指定Webpack的构建行为和优化策略。
  6. DevServer：Webpack的开发服务器，用于提供热更新、自动刷新和代理等开发辅助功能。
  7. Bundle：Webpack的输出结果，由多个模块打包而成，是应用程序的可执行文件。
  8. Chunk：Webpack中的代码块，由多个模块组成，用于实现代码分割和按需加载等优化。
  9. Module：Webpack中的模块，可以是JavaScript、CSS、图片等各种类型的文件，用于构建依赖图和打包输出。

总之，Webpack是一个非常复杂和强大的工具，包含了很多不同的概念和部件，可以帮助前端开发者管理模块依赖关系、优化资源文件大小和加载速度、提高开发效率等。

  • Webpack中Loader与Plugin的区别

    • Loader

    Loader主要用于对模块的源代码进行转换，将不同类型的文件转换成Webpack能够处理的模块。例如，将ES6的代码转换成ES5，将Less或Sass文件转换成CSS等。

    Webpack的Loader机制允许在打包过程中对代码进行处理，并将处理后的代码加入到打包后的输出文件中。每个Loader都只负责一种类型的文件转换，所以可以使用多个Loader将源代码转换成Webpack能够处理的模块。

    • Plugin

    Plugin主要用于在Webpack打包过程中执行一些额外的操作，例如，对打包的输出文件进行优化、压缩、代码分离等操作。Plugin可以用来处理整个打包过程，而不仅仅是单个文件的转换。

    Plugin可以在Webpack的构建生命周期中的不同阶段注入自定义的逻辑，例如，在Webpack打包结束后，生成一个可供部署的zip包，或者在打包完成后将打包文件上传至服务器等。

    总之，Loader和Plugin在Webpack中扮演着不同的角色，Loader主要负责源代码的转换，而Plugin则用于处理Webpack的构建过程。在实际使用中，可以根据需求选择合适的Loader和Plugin来进行开发。

  • 前端为什么需要打包

    前端打包是指将多个前端文件（如 HTML、CSS、JavaScript、图片等）打包成一个或多个文件的过程，以便于在浏览器中加载和运行。

    前端需要打包的原因有以下几点：

    1. 提高加载速度：通过将多个文件打包成一个或几个文件，可以减少网络请求次数，提高页面加载速度。
    2. 模块化管理：打包工具可以将代码分割成不同的模块，使得代码更易于维护和管理。
    3. 转换和优化：打包工具可以对代码进行转换和优化，比如压缩 JavaScript 和 CSS 代码，优化图片大小等，减小文件体积，提高加载速度。
    4. 支持多种格式：打包工具可以将不同格式的文件打包成一个通用的格式，以便于在浏览器中加载和运行。

    综上所述，前端打包可以提高页面性能和开发效率，是前端开发中不可或缺的一环。

  • 如何处理跨域请求

    跨域请求是指在浏览器中向不同源的服务器发送请求。由于浏览器的同源策略限制，跨域请求可能会被浏览器拒绝。以下是一些处理跨域请求的方法：

    • JSONP

    JSONP是一种利用<script>标签的跨域方法。客户端通过在URL中指定回调函数的名称，服务器返回一个JavaScript函数调用，该函数的参数就是服务器返回的数据。由于<script>标签不受同源策略限制，因此可以跨域调用服务器端的脚本并获取数据。但是，JSONP只支持GET请求，且只支持纯文本数据，不支持二进制数据和其他格式的数据。

    • CORS

    跨域资源共享（CORS）是一种在服务端设置的方式，允许客户端跨域请求服务器上的资源。通过在服务器端设置相应的响应头，指定允许哪些域名进行访问，即可实现跨域请求。CORS支持各种HTTP请求，包括GET、POST、PUT等，并且支持各种数据格式，如JSON、XML等。

    • 代理服务器

    代理服务器是一种将客户端请求转发到目标服务器的中间服务器。客户端向代理服务器发出请求，代理服务器再将请求转发到目标服务器。由于请求是在同一个域名下发出的，因此不会遇到跨域问题。代理服务器可以在客户端和目标服务器之间进行数据转换，实现协议转换、数据格式转换等功能。

  • 什么情况下导致跨域问题

    跨域问题指的是在网页中通过 JavaScript 发送 HTTP 请求时，如果该请求的域名与当前网页的域名不一致，就会触发跨域问题。

    当它不一致的时候,发送的请求被浏览器拒绝接收(不是不发请求),

    跨域请求分为两种请求,一种是简单请求,一种是非简单请求

这是由浏览器的同源策略所引起的。

同源策略指的是，一个网站的脚本只能访问来自于同一网站的资源，而不能直接访问来自于其它网站的资源。同源策略是为了保证网络安全而存在的，防止恶意代码从一个网站中获取到用户的敏感信息，例如 cookie 等。

下面是导致跨域问题的几种情况：

  1. 不同域名：例如从 http://example.com 向 http://api.example.net 发送请求。
  2. 不同端口号：例如从 http://example.com:80 向 http://example.com:8080 发送请求。

  3. 不同协议：例如从 http://example.com 向 https://example.com 发送请求。

    当 JavaScript 代码向跨域的资源发送请求时，浏览器会发出一个 OPTIONS 请求，请求服务器返回一个 CORS 头部信息，告诉浏览器该跨域请求是否被允许。如果服务器返回的 CORS 头部信息不允许该请求，则浏览器会阻止该请求

  • Vuex中的$store.state和sessionStorage&localStorage的区别

$store.state 是 Vuex 中的状态管理对象，它是一个响应式的数据仓库，可以用来存储和管理全局共享的数据。$store.state 中存储的数据可以在应用的各个组件中进行访问和修改，所有的修改都是响应式的，即当数据发生变化时，相关的组件会自动更新视图。

sessionStorage 和 localStorage 是浏览器提供的 Web 存储方案。它们都可以用来在浏览器端存储数据，区别在于：

  1. 存储时效：sessionStorage 中存储的数据在当前会话结束时会被清除，而 localStorage 中存储的数据会一直保存在浏览器中，直到被手动清除或过期。

  2. 存储范围：sessionStorage 中存储的数据只能在当前浏览器窗口或标签页中共享，而 localStorage 中存储的数据在同源的所有窗口和标签页中共享。

  3. 数据类型：sessionStorage 和 localStorage 中只能存储字符串类型的数据，如果要存储其他类型的数据，需要进行序列化和反序列化操作。

    sessionStorage 和 localStorage 存储的值必须是字符串形式，因此如果要存储对象、数组等复杂数据类型，需要进行序列化操作将其转化为字符串，存储时需要将序列化后的字符串作为值存储；读取时需要进行反序列化操作将字符串转化为原始的对象或数组形式。

总的来说，$store.state 和 sessionStorage & localStorage 的主要区别在于它们的作用和使用场景不同。$store.state 主要用来管理全局共享的数据，而 sessionStorage 和 localStorage 主要用来在浏览器端存储和管理用户的个性化设置和数据缓存等。

  • v-show 和 v-if 指令的共同点和不同点?

    1. 相同点： v-show 和 v-if 都能控制元素的显示和隐藏。
    2. 不同点： 实现本质方法不同 v-show 本质就是通过设置 css 中的 display 设置为 none，控制隐藏 v-if 是动态的向 DOM 树内添加或者删除 DOM 元素
    3. 编译的区别 v-show 其实就是在控制 css v-if 切换有一个局部编译/卸载的过程，切换过程中合适地销毁和重建内部的 事件监听和子组件
    4. 编译的条件 v-show 都会编译，初始值为 false，只是将 display 设为 none，但它也编译了 v-if 初始值为 false，就不会编译了
    5. 性能比较 v-show 只编译一次，后面其实就是控制 css，而 v-if 不停的销毁和创建，故 v-show 性能更好一点。 注意点：因为 v-show 实际是操作 display:” “或者 none，当 css 本身有 display：none 时，v-show 无法让显示 总结（适用场景）：如果要频繁切换某节点时，使用 v-show（无论 true 或者 false 初始都会进行渲染，此后通过 css 来控制显示隐藏，因此切换开销比较小，初始 开销较大），如果不需要频繁切换某节点时，使用 v-if（因为懒加载，初始为 false 时，不会渲染，但是因为它是通过添加和删除 dom 元素来控制显示和隐藏的， 因此初始渲染开销较小，切换开销比较大）

  • 为什么避免 v-if 和 v-for 用在一起

    • 当 Vue 处理指令时，v-for 比 v-if 具有更高的优先级，通过 v-if 移动到容 器元素，不会再重复遍历列表中的每个值。取而代之的是，我们只检查它一次， 且不会在 v-if 为否的时候运算 v-for。

      • 在 vue 2.x 中，在一个元素上同时使用 v-if 和 v-for 时，v-for 会优先作用。

      • 在 vue 3.x 中，v-if 总是优先于 v-for 生效。

  • JS异步执行流程

    JavaScript 是单线程执行的语言，也就是说在同一时刻只能执行一个任务。但是，它提供了异步编程的机制来处理长时间运行的操作，例如网络请求、文件读取、定时器等，以避免阻塞主线程的执行。

    在 JavaScript 中，异步操作通常是通过回调函数、Promise 或 async/await 进行处理的。下面是 JavaScript 中异步执行的基本流程：

    1. 执行同步代码，遇到异步代码时，将其挂起，继续执行后面的同步代码；
    2. 将异步代码添加到任务队列（Task Queue）中，等待执行；
    3. 当任务队列中的异步代码准备就绪（如网络请求返回数据、定时器到期等），就会被添加到执行栈（Execution Context Stack）中等待执行；
    4. 执行异步代码，并将其从执行栈中移除；
    5. 回到步骤 1，继续执行后面的同步代码，直到所有代码执行完毕。

    需要注意的是，任务队列中的异步代码不一定会按照添加顺序依次执行，而是会根据其优先级进行调度执行。在浏览器环境下，异步代码的优先级通常是按照以下顺序进行调度：

    1. 宏任务（macrotask）：例如 setTimeout、setInterval、XMLHttpRequest 等；
    2. 微任务（microtask）：例如 Promise、MutationObserver 等；
    3. requestAnimationFrame。

    在每个宏任务执行完毕后，会先执行所有的微任务，然后再进入下一个宏任务。这也就是常说的“先微任务，后宏任务”的执行顺序。

    • 异步代码添加到任务队列是添加到宏任务队列还是微任务

      异步代码添加到任务队列时，可能会被添加到宏任务队列中，也可能会被添加到微任务队列中，这取决于异步代码的类型。

      宏任务（macro task）是指需要在当前任务执行完毕后立即执行的任务，例如定时器回调函数、网络请求的回调函数、DOM 事件处理函数等。当异步代码完成后，会被添加到宏任务队列中，等待主线程执行完当前的执行栈后，再从宏任务队列中取出一个宏任务执行。

      微任务（micro task）是指在当前任务执行完毕后，需要立即执行的任务。例如 Promise 的 then/catch 函数、MutationObserver 的回调函数等。当异步代码完成后，会被添加到微任务队列中，等待当前宏任务执行完毕后立即执行。

      需要注意的是，微任务优先级比宏任务高，即在主线程执行完当前执行栈后，会首先执行微任务队列中的所有任务，直到微任务队列为空，才会继续执行宏任务队列中的任务。因此，在处理异步代码时，需要注意任务队列的顺序。

  • Vue2双向数据绑定和Vue3的数据响应式的区别

    ​ Vue2使用的是双向数据绑定，也就是当数据发生变化时，视图会自动更新，同时当视图发生变化时，数据也会自动更新。这个机制是通过Object.defineProperty()和getter/setter来实现的。在Vue2中，每个组件实例都有一个相应的watcher对象，它会监听组件内所有数据的变化，并且在数据变化时触发组件的重新渲染。

    ​ Vue3中引入了新的响应式系统，使用了ES6的Proxy对象来实现。相对于Vue2的双向绑定机制，Vue3的响应式系统更加灵活，能够更细粒度地跟踪数据的变化，并且提供了更好的性能。在Vue3中，当数据发生变化时，只有与该数据相关联的组件会被重新渲染，而不是整个组件树。这个特性在大型应用中非常有用，因为它可以显著提高应用的性能。

    ​ 总的来说，Vue3的响应式系统相对于Vue2的双向数据绑定机制更加灵活和高效，但是需要注意的是，在使用Vue3时，需要使用ref和reactive等新的API来声明响应式数据，而不是使用Vue2中的data属性。

  • Vue中如何通信

    1. Props 和 Events：父组件通过 props 向子组件传递数据，子组件通过事件（Event）向父组件传递数据。这是 Vue 中最常见的通信方式。
    2. Vuex：Vuex 是 Vue 官方提供的状态管理库，可以在不同组件之间共享数据。它采用集中式存储管理应用的所有组件的状态。
    3. $emit 和 $on：Vue 中的实例对象 $emit() 方法可以向父级组件发送事件，并且在父级组件中使用 $on() 方法监听该事件。
    4. $parent 和 $children：通过 $parent 属性可以访问父级组件，通过 $children 属性可以访问子级组件。这种方式不太常用，因为在组件层级嵌套较深时，会很难维护。
    5. $refs：通过 $refs 属性可以访问组件实例或者原生 DOM 元素，可以在不同组件之间进行通信
    6. EventBus(中央事件总线):创建一个新的Vue实例，以后它就承担起了组件通信的桥梁，也就是中央事件总线
    7. 依赖注入:provide 和 inject 可以帮助我们解决这一问题。 [1] 一个父组件相对于其所有的后代组件，会作为依赖提供者。任何后代的组件树，无论层级有多深，都可以注入由父组件提供给整条链路的依赖。

  • Vuex工作原理

    • Vuex 是 Vue.js 应用程序开发的状态管理库。它采用了 Flux 架构，它通过一个中心化的状态管理器来管理所有组件的状态。

      Vuex 的工作原理可以概括为以下几个步骤：

      1. 状态存储：Vuex 将所有组件的状态存储在一个中央存储库中，称为“store”。
      2. 组件访问：组件可以通过在其计算属性或方法中引用 store 中的状态来访问 store 中的数据。
      3. 修改状态：组件不能直接修改存储在 store 中的状态。相反，它们必须分发“actions”，actions 是一种提交更改请求的方式。
      4. 执行 mutations：当组件分发一个 action 时，Vuex 将其转发给 store 的“mutations”（变化）部分。mutations 是 Vuex 中用于修改状态的方法。
      5. 状态更新：当一个 mutation 执行时，它会修改 store 中的状态。当 store 的状态改变时，所有订阅 store 的组件都将自动更新。

      总之，Vuex 的工作原理是通过一个中央存储库来管理所有组件的状态，并通过 actions 和 mutations 来修改和更新状态。这种状态管理模式可以帮助开发人员更好地组织应用程序的状态，并简化应用程序的调试和维护。
